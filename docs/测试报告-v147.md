## 本次变更

1. **修复 CloudBase `set/update` 返回成功但云端数据为空的问题**
   - 分析 CloudBase v2 JS SDK 源码发现：`doc.set()` 内部调用 `database.updateDocument`，对固定 ID 文档存在写入异常；`doc.update()` 同样不可靠。
   - `pushToCloud()` 改为先 `docRef.remove()` 删除旧固定 ID 文档，再 `docRef.create(docData)` 创建新文档（内部调用 `database.addDocument`）。
   - 若固定 ID `create` 失败，则回退到 `collection.add(docData)` 创建随机 ID 文档，并在文档中写入 `docId: 'finance_data'` 字段。
2. **增强拉取回退逻辑**
   - `pullFromCloud()` 优先尝试固定 ID 读取；失败或无数据时，通过 `collection.where({ docId: 'finance_data' }).get()` 查询随机 ID 文档，按 `updatedAt` 排序取最新。
3. **增强推送后验证与旧文档清理**
   - 推送后固定 ID 读取验证；失败时通过 where 查询验证；
   - 随机 ID 模式下保留最近 2 条文档并清理旧文档，避免集合无限膨胀。
4. **版本号统一升级到 v147**
   - `index.html` / `service-worker.js` / `js/storage.js` / `tests/smoke_test.js` 全部同步。

## 根因说明

v145/v146 日志显示：`doc.update()` / `doc.set()` 返回成功（`updated: 1` 或 `upsertedId: finance_data`），但随后 `get()` 回读发现云端文档为空或仅有 `_passwordHash:null`/`_passwordEnabled:false`，业务数据 `jsonData`/`data` 不存在。这说明 CloudBase v2 SDK 对固定 ID 文档的 `updateDocument` 调用存在不可靠的写入行为。通过改用 `remove()` + `create()`（`addDocument`）绕开该问题，并在 `create` 失败时回退到 `collection.add()` 创建随机 ID 文档，确保写入路径至少有一条可用。
